# 生产链路锁定区

本文件定义仓库中最不能乱动的生产链路。任何 AI 工程师修改这些区域前，必须先说明风险、范围和回滚办法，并由零号员工审计。

## 1. 最高原则

生产链路稳定性高于任何单次优化。没有明确需求、没有风险说明、没有复查和验证，不得修改生产链路。

## 2. P0 绝对敏感区

以下内容不得随意修改：

- PAT、GH_PAT、GITHUB_TOKEN、secrets。
- Telegram token、chat_id、发送逻辑。
- 主入口文件和运行入口。
- workflow 触发入口、定时任务、权限、并发组。
- 缓存 restore/save、kline_cache 缓存闭环。
- 数据门控：15:00 数据门控、DATA_GATE_TARGET_DATE、LAST_TRADE_DAY。
- BaoStock 登录、重登、逐票数据校验和 fallback。
- artifact 输出路径和报告主结构。

## 3. P1 高风险区

以下内容可以改，但必须审计：

- schedule 频率，尤其是高频定时。
- workflow_run 链式触发。
- 自动提交逻辑。
- 任何会导致重复推送、递归触发、无限循环的逻辑。
- wrapper、补丁、临时兜底。
- 大面积重构主脚本。

## 4. P2 必须记录区

以下内容改动后必须写入工程维护手册：

- 报告格式变化。
- 文档入口变化。
- 成功经验变化。
- 失败教训变化。
- 员工职责变化。
- 新员工、新战法、新 workflow。

## 5. 修改前检查清单

修改生产链路前必须回答：

- 为什么必须改？
- 是否有更小范围的改法？
- 是否影响一号到五号正式任务？
- 是否会导致重复触发？
- 是否会导致高频占用 runner？
- 是否会破坏缓存、推送、数据门控或 artifact？
- 如何验证？
- 如何回滚？

## 6. 修改后验收

- 有 commit sha。
- 已复查实际文件。
- 已验证 workflow 或明确待验证。
- 已写入工程维护手册。
- 已由零号员工审计或等待零号员工审计。
