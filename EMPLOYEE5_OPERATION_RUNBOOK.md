# 五号员工运行总手册

本文档用于让后续 AI 员工、AI 工程师、维护者快速理解五号员工的身份、模型思路、报告口径、自动触发机制、成功标准、失败排查方式和禁止事项。任何后续优化五号员工的人，必须先阅读本文件，再改代码。

---

## 1. 五号员工身份

五号员工不是买入推荐员工，也不是明天追涨停的交易员工。

五号员工的身份是：

- 涨停样本研究员。
- 极强样本归因研究员。
- 强势结构考古员。
- 经验归纳员。
- 给一号、二号、三号、零号员工提供战法素材和验证样本的数据员工。

五号员工每天研究全市场涨停股，目标不是解释“它今天为什么涨停”，而是长期归纳：

- 什么样的结构更容易涨停。
- 什么样的量能更容易涨停。
- 什么样的周期位置更容易涨停。
- 什么样的洗盘过程更容易涨停。
- 什么样的试盘行为更容易涨停。
- 什么样的筹码路径更容易涨停。
- 什么样的市场环境更容易形成涨停、连板或井喷式上涨。

五号员工输出的是研究样本，不是直接买入指令。

## 2. 用户明确要求

### 2.1 全市场涨停必须全量覆盖

五号员工必须覆盖 A 股所有涨停样本，包括：

- 主板。
- 中小板。
- 创业板。
- 科创板。
- 北交所。
- ST / 非 ST 样本。

注意：不同板块涨停幅度不同，不能用同一个 10cm 逻辑判断所有股票。

当前涨停阈值原则：

- ST：约 5%。
- 主板 / 中小板：约 10%。
- 创业板 / 科创板：约 20%。
- 北交所：约 30%。

用户曾明确指出：某交易日外部口径涨停约 136 只，而旧版五号员工只识别 115 只，主要疑点是北交所未完全纳入。因此后续任何版本必须把北交所作为硬校验项，不能只依赖普通涨停池。

### 2.2 普通涨停只做统计，不要堆股票名单

Telegram 主报告不要把 100 多只涨停股票名字全部列出来。

普通涨停样本应该输出大类统计，例如：

- 主板涨停：多少只。
- 创业板涨停：多少只。
- 科创板涨停：多少只。
- 北交所 30cm 涨停：多少只。
- 10cm 涨停：多少只。
- 20cm 涨停：多少只。
- 30cm 涨停：多少只。
- 突破 20 日 / 月线窗口高点：多少只。
- 突破 60 日 / 季度窗口高点：多少只。
- 20 日涨幅超过 50%：多少只。
- 60 日涨幅超过 100%：多少只。
- 健康放量涨停：多少只。
- 爆量分歧涨停：多少只。
- 缩量快速板：多少只。
- 高位加速样本：多少只。
- 低位启动样本：多少只。

完整股票名单可以保留在 artifact 的 JSON 文件中，不要塞进 Telegram 主报告。

### 2.3 每天只选 3 只重点深度样本

五号员工不需要对全部涨停股做深度分析。

每天从涨停池中挑 3 只“周期性大涨 + 当日涨停”的重点样本做深度分析。

优先选择：

- 最近 20 日涨幅超过 50%。
- 最近 60 日涨幅超过 100%。
- 最近 100 日涨幅超过 150%。
- 连续涨停。
- 井喷式上涨。
- 主升浪明显。
- 当日再次涨停。

这 3 只样本是用来研究“爆发式上涨股票到底怎么走到今天的”，不是让用户第二天直接买。

### 2.4 每只深度样本至少 30 个独立维度

每只深度样本必须至少输出 30 条归因观察。

这 30 条不能来自同一个维度，不能把同一件事拆成 30 句凑数。

必须是 30 个不同思路、不同观察角度、不同证据来源的独立维度。

每条归因必须尽量具备：

- 可观察。
- 可复盘。
- 可验证。
- 有结构含义。
- 有历史路径含义。
- 有后续验证价值。

不能写空话，例如：

- “走势较强”。
- “资金关注”。
- “形态不错”。
- “可能有资金”。

这些不能计入 30 条。

### 2.5 深度分析应该从大周期历史路径开始

对爆发式、井喷式上涨股票，不能只看当天涨停。

应尽量从大周期往小周期梳理：

- 多年前的大级别平台。
- 年线 / 250K 结构。
- 100K 结构。
- 60K / 季度结构。
- 20K / 近似月线结构。
- 历史最大量 K。
- 历史筹码密集区。
- 左侧压力带。
- 试盘。
- 洗盘。
- 缩量。
- 平量。
- 再试盘。
- 当日涨停触发。

用户要的是“周期性的历史性分析”，不是表层归因。

## 3. 五号员工报告结构

五号员工 Telegram 报告应采用两层结构。

### 3.1 第一部分：全市场涨停大类统计

只输出统计，不堆股票名。

必须包括：

- 日期。
- 涨停总识别数量。
- 板块分布。
- 来源分布。
- 北交所数量。
- 历史 K 线失败数量。
- 大类标签统计。

示例：

```text
🧬【五号员工-涨停板归因】
日期：20260522
涨停总识别：136只
板块分布：{"主板":70,"中小板":30,"创业板":11,"科创板":4,"北交所":21}
来源分布：{"zt_pool":115,"bj_spot_em":21}
历史K线失败：0只

一、全市场涨停大类统计：
- 10cm涨停：100只
- 20cm涨停：15只
- 30cm涨停：21只
- 北交所30cm弹性样本：21只
- 健康放量涨停：X只
- 突破20日/月线窗口高点：X只
- 20日涨幅超50%：X只
```

### 3.2 第二部分：3只周期性大涨深度样本

每只深度样本应包含：

- 股票名称和代码。
- 所属板块。
- 20 / 30 / 60 / 100 日涨幅。
- 30+ 跨维度归因观察。
- 后续验证点。

示例结构：

```text
二、今日3只周期性大涨深度样本：

【深度样本1】某某股份(000000) 主板
20/30/60/100日涨幅：{"20d":58.3,"30d":72.1,"60d":128.5,"100d":166.2}
30+跨维度归因观察：
1. 【大周期】250日区间分位=...
2. 【核心线】收盘价相对250日左侧高点...
3. 【历史筹码】近250日最大成交量...
...
30. 【后续验证】T+20验证结构持续性。
```

## 4. 深度样本 30 个维度池

深度样本必须跨维度覆盖，不能单一维度堆叠。

建议至少覆盖以下 16 类中的 10 类以上：

1. 大周期路径：250K、100K、60K、20K 固定K聚合结构。
2. 核心线/压力带：历史核心压力、实体顶底、影线共振、筹码集中带。
3. 历史筹码：历史最大量K、次大量K、放量阳/阴K、筹码交换区。
4. 形态结构：平台、箱体、凹口、双峰、双肩、头肩、圆弧、破底翻。
5. 趋势结构：下跌趋势线、上升趋势、台阶抬升、重心上移。
6. 洗盘蓄势：洗盘时长、横盘时长、缩量、平量、假跌破、回踩不破。
7. 波动率压缩：振幅收敛、K线变窄、极端波动减少、爆发前夜压缩。
8. 量能演化：量比、标准倍量、极端爆量、平台均量、量能波动率、阳量阴量。
9. 试盘行为：上影试盘、放量冲高回落、假突破、二次突破、前次失败高点。
10. 供应吸收：左侧套牢区测试、压力区多次攻击、供应消耗、压力转支撑。
11. 当前触发：涨停当天突破什么线、是否打穿结构带、是否站稳关键位。
12. 当前K线质量：实体、收盘位置、上影线、跳空、一字、烂板、量价效率。
13. 位置与过热：低位/中低位/高位、MA5/MA10距离、长期区间分位。
14. 板块与市场：板块扩散、连板高度、涨跌停家数、风险偏好、题材一致性。
15. 风险与反证：ST/北交所特殊性、缺少换手、上方大压力、回踩未验证、失败反证。
16. 后续验证：T+1/T+3/T+5/T+8/T+13/T+20 观察点。

## 6. 自动触发机制

五号员工支持三种触发方式。

### 6.1 定时触发

workflow 中设置工作日定时运行。

历史设定为北京时间晚间运行，需注意 BaoStock / AKShare 数据稳定性。用户曾明确指出，某些数据源太早拉 K 线可能不稳定，最好等数据稳定后再跑。

### 6.2 GitHub Actions 手动触发

用户可以在 GitHub 页面：

```text
Actions → 五号员工 → Run workflow
```

手动触发。

### 6.3 固定 Issue + 固定标签触发

为了让 AI 助手可以自动触发，当前采用：

```text
固定 Issue #2
固定标签 run-employee5
```

触发步骤：

1. 从固定 Issue #2 移除标签 `run-employee5`。
2. 再给固定 Issue #2 添加标签 `run-employee5`。
3. workflow 监听 `issues.labeled` 事件。
4. GitHub Actions 启动五号员工。
5. 运行名显示为 `五号员工`。
6. 报告推送到 Telegram。

当前 workflow 应尽量只监听：

```yaml
issues:
  types:
    - labeled
```

不要同时监听 `edited`、`unlabeled`、`issue_comment`，否则容易出现多个运行记录，看起来混乱。

## 7. 单实例并发锁

五号员工必须永远只允许一个实例运行。

workflow 必须保留：

```yaml
concurrency:
  group: fifth-employee-singleton-main
  cancel-in-progress: true
```

这表示：

- 如果误触发多次，GitHub 会取消旧任务。
- 只保留最新一次。
- 避免同时两个五号员工抢接口、重复推送。

如果未来出现两个运行记录：

- 一个可能是旧任务正在被取消。
- 一个是新任务排队或运行。
- 最终只应保留最新任务成功完成。

## 8. 成功运行标准

一次合格的五号员工运行，必须满足：

1. Actions 页面显示名称为 `五号员工`，不是 memo 或英文临时名。
2. 只有一个最终成功的五号员工 run。
3. Telegram 收到 `🧬【五号员工-涨停板归因】` 报告。
4. 报告涨停总数应尽量接近全市场外部口径，例如用户指出 20260522 应约 136 只，而不是只显示 115 只。
5. 报告中板块分布必须包含北交所，如果当天北交所有涨停。
6. 报告中普通涨停只显示大类统计，不堆大量股票名。
7. 报告中应有 3 只深度样本。
8. 每只深度样本应有 30+ 跨维度归因观察。
9. artifact 中应生成完整 JSON 和 Markdown 报告。
10. 如果深度样本为空，报告必须明确说明原因，不能伪造深度归因。

## 9. 常见问题与排查

### 9.1 涨停数量只有 115，不是全量 136

优先检查北交所是否纳入。

必须检查：

- `stock_bj_a_spot_em` 是否可用。
- `stock_zh_bj_a_spot` 是否可用。
- 北交所代码 920 / 8 / 4 开头是否正确识别。
- 北交所涨停阈值是否按 30cm 处理。
- 来源分布是否只有 `zt_pool`，如果只有 `zt_pool`，说明北交所兜底没有生效。

### 9.2 深度样本为空

通常原因：历史 K 线接口失败或被限流。

旧版问题是对所有 100 多只涨停股逐只拉历史 K 线，接口容易断连。

正确做法：

- 普通涨停不应逐只拉历史 K 线。
- 先用涨停池和快照做大类统计。
- 深度样本只对候选股票拉历史 K 线。
- 拉够 3 只可分析深度样本后停止。
- 失败数量写入报告或 artifact。

### 9.3 GitHub Actions 日志满屏英文

这是 AKShare / requests 抛出的底层错误，例如：

```text
Connection aborted / Remote end closed connection without response
```

含义：数据源断开连接，通常是接口限流、网络断连或请求过快。

五号员工不应让这些英文错误刷屏。

正确做法：

- 单票历史 K 线失败不逐条打印完整英文异常。
- 只累计失败数量。
- 只保留少量失败样例。
- 最终用中文输出：`历史K线失败：X只，已跳过`。
- 详细失败列表放 artifact，不放 Telegram。

### 9.4 Telegram 报告太长被截断

Telegram 单条消息有长度限制。

正确做法：

- 主报告分段发送。
- 普通涨停只做统计。
- 完整名单放 JSON artifact。
- 深度样本可以分段推送，但不能把普通股票名全塞进主报告。

### 9.5 触发后出现两个五号员工

原因可能是 workflow 同时监听了多个 Issue 事件，例如 `edited`、`labeled`、`comment`。

修复原则：

- 只保留 `issues.labeled` 作为 AI 自动触发口。
- 保留 concurrency singleton。
- 触发时只做“移除标签 → 添加标签”。
- 不再更新 Issue body。
- 不再创建新 Issue。
- 不再评论触发。

## 10. 日志规范

Actions 日志应简洁，尽量中文。

推荐阶段日志：

```text
五号员工开始运行
[1/4] 拉取全市场涨停池...
[2/4] 生成涨停大类统计...
[3/4] 挑选3只深度样本并拉取历史K线...
[4/4] 生成报告并推送...
历史K线失败：X只，已跳过
五号员工运行完成
```

不要刷屏：

- 不要每只股票失败都 print 一大段英文异常。
- 不要把 100 多只股票名称都打印在日志里。
- 不要把敏感环境变量打印出来。

## 11. 数据源原则

五号员工当前使用 AKShare 为主。

### 11.1 涨停池数据源

推荐组合：

- `stock_zt_pool_em`：主涨停池。
- `stock_zt_pool_st_em`：ST 涨停池。
- `stock_zt_pool_previous_em`：连板/昨日涨停池辅助。
- `stock_bj_a_spot_em`：北交所兜底。
- `stock_zh_bj_a_spot`：北交所备用兜底。
- `stock_zh_a_spot_em`：全 A 快照兜底。

### 11.2 历史 K 线数据源

当前主要使用：

- `stock_zh_a_hist` 前复权。
- `stock_zh_a_hist` 不复权。

注意：不要对全部涨停股逐只拉历史 K 线，否则很容易触发接口断连。

### 11.3 成交量优先原则

用户明确要求：筹码核心带、量能分析优先使用成交量或换手率，不优先使用成交额。

原因：成交额受价格高低影响大，底部价格和高位价格不可直接比较；成交量/换手率更适合看筹码交换。

## 12. 后续工程优化建议

### 12.1 把推送逻辑从主模型拆出

为了减少安全弹窗和敏感字段误伤，建议未来拆成：

```text
employee5_runner.py       只负责涨停归因和生成报告
telegram_sender.py        固定不动，只负责读取报告并推送
fifth_employee.yml        固定不动，只负责运行链路
```

后续优化模型时只改 `employee5_runner.py` 的归因逻辑，尽量不碰 workflow 和推送脚本。

### 12.2 建立样本库

五号员工不应只生成日报，还应逐步沉淀样本库：

```text
employee5_reports/limit_up_research_report.json
employee5_reports/limit_up_deep_samples.json
employee5_samples/history.jsonl
```

后续零号员工可验证：

- T+1 是否站稳。
- T+3 是否不跌回结构线。
- T+5 是否继续扩散。
- T+8 是否形成二次承接。
- T+13 / T+20 是否证明该结构有持续性。

### 12.3 对其他员工的输出

五号员工最终应向其他员工提供机器可读反馈：

给一号员工：

- 哪些民间战法结构近期在涨停样本中高频出现。

给二号员工：

- 哪些华尔街式市场结构、周期结构、量能结构在强势样本中有效。

给三号员工：

- 哪些强势样本不能追，哪些适合等待回踩确认。

给零号员工：

- 哪些新规律需要 T+1/T+3/T+5/T+8/T+13/T+20 验证。

## 13. 禁止事项

五号员工维护时禁止：

- 把五号员工做成直接买入推荐系统。
- 把当天涨停原因写成单一故事。
- 用空数据伪造深度分析。
- 把 100 多只股票名称塞进 Telegram。
- 漏掉北交所。
- 把 10cm、20cm、30cm 涨停混成同一标准。
- 对全部涨停股逐只暴力拉历史 K 线。
- 日志刷屏英文底层错误。
- 同时触发多个五号员工。
- 随便新建一堆临时 Issue 触发。
- 修改 PAT、secrets、凭证、Telegram token 本体。
- 自动把未验证规律写入一号/二号/三号生产模型。

## 14. 一句话总结
