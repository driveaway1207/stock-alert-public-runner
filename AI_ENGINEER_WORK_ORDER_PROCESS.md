# AI 工程师工作单制度

本文件定义所有 AI 工程师的标准工作流。后续任何工程修改都必须先建立工作单思维，再动代码或文档。

## 1. 为什么需要工作单

员工越多，越容易出现口头说改好、实际没落地、后续无人能复盘的问题。工作单的目标是让每次修改都可追踪、可复查、可验证、可归档。

## 2. 工作单编号

每次工作建议使用如下编号：

```text
WORK-YYYYMMDD-序号
```

例如：

```text
WORK-20260524-001
```

## 3. 开工前必须写清楚

- 用户原始需求是什么。
- 本次要改什么。
- 本次不改什么。
- 涉及哪些员工。
- 涉及哪些文件。
- 是否涉及 workflow。
- 是否涉及生产链路。
- 验收标准是什么。

## 4. 修改后必须写清楚

- 实际修改了什么。
- 实际修改路径。
- commit sha。
- 是否已复查 GitHub 实际文件。
- 是否已验证 workflow 或脚本运行。
- 是否存在失败尝试。
- 是否存在猴子代码风险。
- 是否已写入工程维护手册。

## 5. 成功标准

一项工作只有同时满足以下条件，才能说“已落地”：

- 有 commit sha。
- GitHub 实际文件复查通过。
- 需要运行的 workflow 或脚本已经验证，或明确标记为待验证。
- 工程维护手册已经记录。
- 失败尝试已经记录。

不能把“已提交”说成“已验证”。不能把“尝试过”说成“已落地”。

## 6. 标准记录模板

```text
## WORK-YYYYMMDD-001｜标题

- 用户需求：
- 本次范围：
- 不做范围：
- 涉及员工：
- 涉及文件：
- 验收标准：
- commit sha：
- 已提交：
- 已复查：
- 已验证：
- 猴子代码风险：
- 失败尝试：
- 归档位置：
```
